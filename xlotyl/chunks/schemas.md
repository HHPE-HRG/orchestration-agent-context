```md
# Shell bridge v1 (Control Plane shell handshake)

For agents: this folder is the **versioned JSON Schema** for the thin envelope carried in `POST /api/ai/query` under `context.shell_bridge`. It is **not** a control-plane orchestration language; Control Plane’s task-card and related schemas remain internal.

## Transport

- **Ingress:** `POST /api/ai/query` on `services/api-service`.
- **Field:** `context.shell_bridge` must match `shell-bridge-envelope.schema.json` when present. If the key is absent, the request is a normal web/API caller (no bridge validation).
- **Governed helper input:** `governed-shell_bridge-envelope-input.schema.json` is the XLOTYL-owned adapter contract used by bridge helpers before they normalize into the wire envelope.

## Idempotency (Phase 1)

Shell-originated turns **must** include `idempotency_key` (required in the envelope schema).

- **Hit:** same `idempotency_key` and identical **idempotency payload hash** → API returns the **cached JSON body** from the prior successful turn (HTTP 200, same shape as a fresh orchestration response).
- **Conflict:** same `idempotency_key` and a **different** payload hash → HTTP **409** with a typed error (`idempotency_key_conflict`). The server does **not** append silently to an existing run.
- **Miss:** unknown key → normal orchestration; on success the response is stored keyed by `idempotency_key` with a TTL (default 24h server-side).
- **Redis unavailable:** replay is skipped (no duplicate suppression); conflicts are still detected if a record appears later.

The payload hash is computed server-side from a canonical encoding of the full `QueryRequest` fields that influence orchestration (see `src/shell_bridge/idempotency.py`).

## Attachments (v1)

See `shell-bridge-envelope.schema.json` `$defs.attachment`. Summary:

- **References:** `kind: "url"` with `https` only (Control Plane may tighten host allowlists later), or `kind: "storage_ref"` with an opaque `ref` string.
- **Inline:** `kind: "inline_base64"` with `media_type` + base64 `data`; **total decoded inline bytes per turn** must not exceed **65536** (64 KiB). Oversize requests fail validation (HTTP 400) with a typed error—**no silent truncation**.

## Streaming (Phase 3+)

Typed events are defined in `events/stream-event.schema.json`. The primary UX stream must **not** be raw workflow logs; use discriminated `type` values such as `run.started`, `run.completed`, etc.

For governed sessions, XLOTYL also projects authoritative workflow lifecycle into **gateway-compatible** session envelopes:

- `GET /api/shell_bridge/sessions/stream` — SSE projection of `sessions.changed` / `session.message`
- `GET /api/shell_bridge/sessions/events` — finite cursor polling over the same projected envelopes

These are projections of governed XLOTYL state, not a second orchestration authority.

## MCP as primary bridge

Cross-exposing Control Plane MCP tools as the **primary** shell path is explicitly **Phase 4** (see `docs/external-orchestration-interfaces.md` in the Control Plane repo). Phase 1 uses HTTPS + this envelope only.
```

```md
# Shell bridge v1 — typed stream events

JSON Schema: [`stream-event.schema.json`](stream-event.schema.json).

## Transport

- **Producer (today):** `POST /api/ai/query/stream` on `services/api-service` returns `text/event-stream`. Each event is one SSE `data:` line containing a single JSON object (no pretty-printed multi-line payloads).
- **Consumer:** external shells and adapters should use a streaming HTTP reader (for example Node `fetch` with a streaming body) and validate each `data:` line against this schema.

## Envelope

Every wire event includes:

| Field       | Meaning |
|------------|---------|
| `type`     | Discriminator (`run.started`, `run.completed`, …). |
| `version`  | Envelope semver string (currently `1.0.0`). |
| `event_id` | Monotonic string counter for this HTTP response (best-effort; suitable for logs). |
| `cursor`   | Optional opaque cursor string (reserved for future replay; producers may omit). |
| `ts`       | **Canonical** UTC timestamp (`date-time`). |
| `timestamp` | Deprecated alias for `ts`; do not emit in new producers. |

Type-specific fields may appear at the top level (for example `workflow_id` on `run.completed` and optional `workflow_id`/`run_id` on `run.progress`) **and** inside `payload`. Producers should keep **governed facts** out of ad-hoc strings; prefer structured `payload` for new fields.

## Ordering and reconnect

- Ordering is **best-effort monotonic** by `event_id` within a single SSE response body.
- **Resume hints (opaque, no durable replay yet):** Clients may send **`Last-Event-ID`** (standard SSE header) and/or query **`event_cursor`** on `POST /api/ai/query/stream`. When either is non-empty, Control Plane emits a leading **`resume.ack`** event echoing those inputs (and `engineering_session_id` from `context` when present), then runs the normal MVP sequence. **This does not replay skipped events** — there is no server-side event buffer; each request still executes `execute_ai_query_pipeline` once unless you add idempotency/stream replay separately.
- **Durable cursor / `Last-Event-ID` replay** is explicitly **out of scope** for the original bridge MVP.

Discovery (why mid-run events are coarse today): see [`docs/ui-integration-v0.md`](../../../../docs/ui-integration-v0.md) and [`docs/operators/xlotyl-stoneforge-runbook.md`](../../../../docs/operators/xlotyl-stoneforge-runbook.md).

## Contract rules

1. **No raw LangGraph / orchestrator log lines** as the primary UX contract — only typed events from this schema family.
2. **MVP limitation:** until `agent-platform` exposes incremental lifecycle or a workflow stream, Control Plane emits **`run.started`**, a coarse **`run.progress`** phase event (`phase=executing`), then **`run.completed`** or **`run.failed`**, plus optional tail events such as **`pending_mode_change`** derived from the final JSON snapshot. Mid-run `clarification.required` / `tool_request` appear only when the backend can produce them.
3. **Typed final answer:** when the orchestrator returns a final assistant message, Control Plane includes it as `run.completed.payload.final_response` (string) so UI/agents can consume structured completion output without parsing free-form logs.

## Cancel

Workflow cancel is **`POST /api/ai/workflows/{workflow_id}/cancel`** (proxy to agent-platform). On **HTTP 200**, the JSON body includes:

- **`orchestrator`** — raw agent-platform response (shape upstream-defined).
- **`cancel_ack`** — typed object matching the **`cancel.ack`** branch of this schema (same fields as an SSE `data:` line would use: `type`, `version`, `event_id`, `ts`, `payload.workflow_id`).

## Validation

Golden fixtures live under [`fixtures/`](fixtures/). CI loads each fixture with `jsonschema` (see `services/api-service/tests/test_shell_bridge_stream_event_fixtures.py`).

## Running stream (UI integration)

`GET /api/running/stream` emits lifecycle events across **workflows** and **DevPlane runs**.

- **Transport**: SSE (`text/event-stream`)
- **Resume**: supports `Last-Event-ID` header (preferred), or `event_cursor` query param
- **Event types**: reuses `run.*` plus `resume.ack` and `stream.error`
- **Disambiguation**: `payload.kind` is one of `workflow` or `dev_run`
```
