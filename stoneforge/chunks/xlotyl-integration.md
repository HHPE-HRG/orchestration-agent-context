# Stoneforge ↔ Xlotyl boundary (agent-safe)

This document exists to make the Stoneforge/Xlotyl split **hard to misinterpret** for coding agents working repo-locally.

**Canonical repos (post org-rename):**

- Stoneforge: `HHPE-HRG/stoneforge`
- Xlotyl: `HHPE-HRG/xlotyl`

## Ownership: what Stoneforge owns

Stoneforge is the **execution/orchestration runtime**. It owns:

- Smithy runtime (agent orchestration loop)
- Quarry data layer (durable JSONL source-of-truth + SQLite cache)
- Agent dispatch, worktree isolation, and worker pools
- Merge/recovery flows and stewards
- Operator dashboards / UI
- `sf` CLI surfaces consumed by operators and by Xlotyl contract tests

## Ownership: what Stoneforge does NOT own

Stoneforge does not own Xlotyl control-plane semantics, including:

- Xlotyl planning structure / graph compilation
- Capability package semantics and smoke metadata
- Response-control / wiki as the source of truth
- Model harness routing contracts (profiles, eligibility)

## Integration boundary (most important rule)

In **Xlotyl-controlled runs**:

- Xlotyl owns planning structure (`graph.json`).
- Stoneforge executes imported graphs.
- Stoneforge Director must not autonomously decompose goals when `execution_mode = xlotyl_stoneforge`.

Stoneforge may visualize and schedule tasks, but must not reshape the plan in that mode.

## Safe Stoneforge-only work

You can work in **Stoneforge only** without checking out Xlotyl as long as you treat imported Xlotyl graphs and decision-provider behavior as an **external contract**:

- dashboards, task queue display, worktree management
- runtime scheduling/dispatch improvements that preserve task lifecycle contracts
- `sf` CLI compatibility surfaces
- Xlotyl-tagged Stoneforge contract tests

## Compatibility gates (run before claiming success)

From `stoneforge/` repo root:

```bash
pnpm install
pnpm --filter @stoneforge/smithy build
pnpm --filter @stoneforge/smithy test:xlotyl-daemon-decision
```

Also run:

```bash
pnpm test
```

…if your changes touch core runtime behavior, dispatch semantics, or CLI contracts.

## When you MUST run a paired checkout test

Treat these as cross-repo contract changes (run both repos’ contract suites):

- `sf` CLI JSON stdout shape/fields
- imported graph semantics or on-disk run artifact layout
- task lifecycle fields / dispatch status semantics
- decision-provider interface used by Xlotyl (`xlotyl_stoneforge` mode)

# Stoneforge and Xlotyl

This repository (**Stoneforge**, `HHPE-HRG/stoneforge`) is the **custom orchestration fork** used alongside the **Xlotyl** platform repo (`HHPE-HRG/xlotyl`). Xlotyl remains the Birtha product monorepo (HTTP APIs, DevPlane, MCP/OMA surfaces, CI, wiki); Stoneforge provides the **Smithy / Quarry** runtime, web dashboards, and agent dispatch UX that sit in front of governed graphs.

## Integration surface

- Xlotyl imports and tests Stoneforge contracts from **`@xlotyl/core-dev-services`**, SF-CLI helpers, and CI workflows that call into Stoneforge semantics. The operator runbook in Xlotyl is the day-to-day checklist: [`xlotyl/docs/operators/xlotyl-stoneforge-runbook.md`](https://github.com/HHPE-HRG/xlotyl/blob/main/docs/operators/xlotyl-stoneforge-runbook.md).
- Stoneforge is **not** a git submodule of Xlotyl; both repos are pinned independently and wired through **npm contracts**, shared scripts, and documentation.
