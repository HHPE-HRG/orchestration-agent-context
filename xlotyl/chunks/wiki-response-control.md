# Control Plane unified wiki — authoring contract

This tree is the **single** long-lived knowledge surface for humans and tooling. Only one subtree feeds orchestration JSON generation.

## Zones (path = authority)

| Path | Role |
|------|------|
| `knowledge/wiki/orchestration/{agent-profiles,domain-knowledge-packages,runtime-environments,techniques,theory}/` | **Routing cards.** Each `*.md` file’s JSON front matter is compiled into [`knowledge/response-control/`](../response-control/) JSON arrays consumed by `evaluate_response_control` in the API service. |
| `knowledge/wiki/projects/` | **Non-routing** project narrative (clients, delivery, specs). The compiler **never** reads this directory. |
| `knowledge/wiki/_proposals/` | **Unapproved edit queue**. Proposal files are not canonical truth until approved and promoted. |
| `knowledge/wiki/index.md` | Human-oriented map of the wiki (optional machine lint later). |
| `knowledge/wiki/topic_index.json` | **Generated** machine-readable map of orchestration shards/pages (`scripts/build_wiki_topic_index.py`; run via `make wiki-compile`). |
| `knowledge/wiki/log.md` | Chronological notes and decisions about wiki structure (optional). |

## Orchestration page format

1. File **must** live under `knowledge/wiki/orchestration/<shard>/` where `<shard>` is one of: `agent-profiles`, `domain-knowledge-packages`, `runtime-environments`, `techniques`, `theory`.
2. Begin the file with **JSON** front matter delimited by `---` lines (stdlib-only compile; no YAML dependency).
3. The JSON block **must** include the full machine record for that shard (same fields as the JSON objects in `knowledge/response-control/*.json`), validated by Pydantic models in `services/api-service/src/control_plane/contracts.py`:
   - `agent-profiles` → `AgentProfilePayload`
   - `domain-knowledge-packages` → `DomainKnowledgePackagePayload`
   - `runtime-environments` → `RuntimeEnvironmentPayload`
   - `techniques` → `TechniqueCardPayload`
   - `theory` → `TheoryCardPayload`
4. Optional metadata keys (stripped before validation; must not collide with card fields):
   - `wiki_zone`: must be `orchestration` when present.
   - `wiki_shard`: must match the parent directory name (`agent-profiles`, `domain-knowledge-packages`, etc.).
5. Markdown **after** the closing `---` is free-form (rationale, diagrams, links). It is **not** parsed into JSON and does not affect routing.

## Workflow

1. Edit markdown under `knowledge/wiki/orchestration/`.
2. From the repo root: `make wiki-compile` (or `python scripts/wiki_compile_response_control.py`).
3. Commit **both** the wiki sources and the regenerated `knowledge/response-control/*.json`.
4. CI runs `make wiki-check` to forbid drift between sources and JSON.

Do **not** hand-edit `knowledge/response-control/*.json` on an ongoing basis; treat those files as **build artifacts** of this wiki.

## Proposal queue contract

Unapproved proposals live in `knowledge/wiki/_proposals/*.md` with JSON front matter validated by `wiki-edit-proposal.schema.json`. They must include:

- `unapproved_source: true`
- `caution: "unapproved source, proceed with caution"`
- lifecycle `status` in `PROPOSED | APPROVED | REJECTED | PROMOTED`

Promotions are editor-governed; proposal files do not directly replace canonical wiki content.
Proposal content uses an editable-draft policy in the HTTP surface: bounded `PATCH /api/control-plane/wiki/proposals/{id}` updates are allowed only while status is `PROPOSED`; moderation lifecycle transitions (`approve`/`reject`/`promote`/archive) remain dedicated endpoints.

## HTTP API (control plane)

Agents and operators use `/api/control-plane/wiki/*` (same permission model as the MCP adapter in `scripts/control_plane_mcp_server.py`).

| Scope | Capability |
|-------|------------|
| `wiki:read` | List/read/search orchestration pages; list proposals; validate payloads. |
| `wiki:read_projects` | List/read `knowledge/wiki/projects/**` (non-routing narrative). |
| `wiki:write_orchestration` | Create/update/delete canonical orchestration pages under `knowledge/wiki/orchestration/**`. |
| `wiki:write_projects` | Create/update/delete canonical projects pages under `knowledge/wiki/projects/**`. |
| `wiki:propose` | `POST /wiki/proposals` — create `_proposals/*.md`. |
| `wiki:approve` | Approve/reject queued proposals. |
| `wiki:admin` | Grants all of the above. |

When `wiki_control_plane_auth_enabled` is true, every wiki route requires `X-API-Key` registered in `wiki_api_keys_json` (`user_id` + `permissions`). Approve/reject bind `approved_by` to the authenticated `user_id`.

Git remains authoritative for merged markdown and compiled JSON; see [docs/wiki_git_pr_workflow.md](../../docs/wiki_git_pr_workflow.md).

# Wiki proposal queue

This directory stores unapproved wiki edit proposals produced by orchestration runs.

- File format: markdown with JSON front matter delimited by `---`.
- Front matter payload validates against `wiki-edit-proposal.schema.json`.
- Canonical wiki pages under `knowledge/wiki/orchestration/**` and `knowledge/wiki/projects/**`
  are not edited directly until a proposal is approved and promoted.

Lifecycle:

1. `PROPOSED` — created automatically, marked `unapproved_source: true`.
2. `APPROVED` — explicitly approved by the head editor.
3. `PROMOTED` — applied to canonical wiki and compiled catalogs in a promotion run/PR.
4. `REJECTED` — preserved for audit, excluded from promotion.

# Documentation layers (wiki vs backend ops)

This page is the **canonical policy** for how agents and humans should use Control Plane/Xlotyl documentation. It lives under [`knowledge/wiki/meta/`](.) — non-routing narrative; the response-control compiler does not read this directory.

## Canonical narrative: `knowledge/wiki/`

- **Start here** for product story, orchestration context, and project-facing notes: see [Wiki index](../index.md) and [SCHEMA.md](../SCHEMA.md) for zones (`orchestration/`, `projects/`, `_proposals/`, and this `meta/` tree).
- The wiki is the **single long-lived knowledge surface** for “what exists and how we talk about it” without pulling readers into backend file trees by default.

## Operational truth: `xlotyl/docs/` and services

- **`xlotyl/docs/**`** holds **operational** material: ADRs, matrices, runbooks, workflows tied to how the repo is run and changed. Entry point: [`xlotyl/docs/README.md`](../../../docs/README.md).
- **Service code and deep paths** (`xlotyl/services/**`, etc.) are authoritative for behavior but are **not** the default map for general product work.

**Rule:** The wiki **references** `xlotyl/docs/**` and related paths; it does **not** duplicate or maintain parallel copies of operational prose. When ops docs change, update them at the source and adjust **links** from the wiki if URLs or titles shift—not a second write-up of the same procedure.

## Friction model (session-aware)

We intentionally create **friction** so agents doing general work do not treat backend docs as the primary compass, and **remove friction** when the session is explicitly Xlotyl backend work.

| Mode | When | Behavior |
|------|------|----------|
| **High friction (default)** | General product work; `project=xlotyl` is not the focus | Prefer explanations and links under `knowledge/wiki/**`. Defer deep citations of `xlotyl/docs/**` unless the user needs a specific ADR, matrix row, or runbook step. Do not paste long excerpts from ops docs into the wiki. |
| **Low friction** | Any of: user states XLOTYL backend / infrastructure work; active edits under `xlotyl/**`; optional local marker file (see root [`AGENTS.md`](../../../../AGENTS.md)) | Citing `xlotyl/docs/**`, ADRs, and service paths is appropriate. Still avoid duplicating wiki narrative inside ops-only files. |

This is **not** a hard ban: it is a default posture. Escalate to `xlotyl/docs` whenever procedure or ADR text is the source of truth.

## “Pop the hood”

Use **`xlotyl/docs`**, **ADRs**, and **`xlotyl/services/**`** as first-class when:

- Implementing or debugging backend behavior.
- Changing governed routes, contracts, or deployment.
- The conversation is explicitly about Xlotyl infrastructure or a focused backend sprint.

Use the **wiki first** when explaining behavior to a mixed audience or when the task is not primarily backend surgery.
