# orchestration-agent-context

A generated, agent-readable context mirror of the HHPE-HRG orchestration repos.

This repo is **not** a source-of-truth fork. It is a **compact, searchable, chat-agent-friendly projection** of the working repos so web-based agents can inspect and reason about the codebase without GitHub connector access, large repo uploads, or local checkouts.

## Why this repo exists

Web-based chat agents typically cannot:

- clone a private repo,
- consume a 1+ GB monorepo,
- traverse `node_modules` / `.venv` / build artifacts safely,
- read secrets without risking leakage.

This mirror gives them a small, curated, deterministic view:

- boundary docs (who owns what),
- contract indexes (graph format, capability packages, sf-cli),
- repo maps,
- symbol indexes,
- selected source chunks (markdown-fenced),
- a `source-manifest.json` for freshness tracking.

It is intended to be paginated, grepped, and quoted by agents — not executed.

## Source repos mirrored

The mirror is generated from these canonical repos:

| Local path           | GitHub repo               | Role                                                                                                  |

| `../stoneforge`      | `HHPE-HRG/stoneforge`     | Smithy runtime, Quarry data layer, dispatch, worktrees, dashboards, sf CLI, steward workflows.        |
| `../claw-code`       | `HHPE-HRG/claw-code`      | Vendored coding-agent orchestration dependency.                                                       |

`HHPE-HRG/xlotyl` and `HHPE-HRG/stoneforge` are the **two source-of-truth repos** for the orchestration story. Everything else is supporting context.

## How chat agents should use it

1. Start in [`AGENTS.md`](AGENTS.md). It states the rules of engagement.
2. Read [`integration/00-xlotyl-stoneforge-boundary.md`](integration/00-xlotyl-stoneforge-boundary.md) before recommending any cross-repo change.
3. Read the relevant `00-boundary.md` for the repo you're touching.
4. Use `02-contract-index.md` and `03-symbol-index.md` to locate code without re-cloning.
5. Quote specific chunks under `chunks/` when explaining behavior.
6. Cite paths from this mirror **and** the canonical path inside the source repo (e.g. `xlotyl/chunks/response-control.md` plus `xlotyl/services/response-control-framework/...`).
7. If context looks stale (manifest timestamp > a few days, or a referenced symbol is missing), ask a human to run `./refresh.sh`.

## How humans refresh it

```bash
git clone git@github.com:HHPE-HRG/orchestration-agent-context.git
cd orchestration-agent-context

# Expected sibling layout:
#   ../xlotyl
#   ../stoneforge
#   ../claw-code
#   ../void
#   ../openclaw

./refresh.sh
git status
git add .
git commit -m "refresh agent context"
git push
```

The refresh script:

- discovers tracked files via `git ls-files` per source repo,
- applies the exclusion patterns in [`context-config.json`](context-config.json),
- writes a per-repo file index (`generated/file-index.json`),
- writes a ripgrep-based symbol index (`generated/symbol-index.json`),
- writes domain-grouped markdown chunks under `xlotyl/chunks/` and `stoneforge/chunks/`,
- writes [`generated/source-manifest.json`](generated/source-manifest.json) with commit SHAs, branches, and timestamps.

If `repomix` (`npx repomix`) is installed, the script will use it for chunk generation. Otherwise it falls back to a simple shell chunker that produces the same on-disk shape.

## What is excluded

The mirror is deliberately conservative. The following are **never** committed:

- `.git/`, `node_modules/`, `.venv/`, `dist/`, `build/`, `coverage/`, `.cache/`, `__pycache__/`
- raw `.env*` files, anything matching `secrets*`
- private keys, model weights, dataset binaries
- binary media (`*.png`, `*.jpg`, `*.jpeg`, `*.webp`, `*.gif`, `*.mp4`, `*.mov`)
- archives (`*.zip`, `*.tar`, `*.gz`)
- non-source `*.lock` files unless explicitly needed

Exclusion is enforced in two places:

1. `git ls-files` already filters out `.gitignore`'d junk in the source repos.
2. `refresh.sh` applies a hard pattern blocklist on top.

Before commit, `refresh.sh` runs a secret scan with ripgrep and aborts if anything plausibly secret-looking ends up under `chunks/` or `generated/`.

## How to verify freshness

```bash
jq '.' generated/source-manifest.json
```

Inspect:

- `generated_at` — should be recent.
- `repos[].commit_sha` and `repos[].branch` — should match the source repo HEADs you expect.
- `repos[].dirty` — `true` means the source checkout had uncommitted changes when the mirror was generated. The mirror still reflects working-tree state; just be aware.


## Out of scope

- This repo does **not** run the product. It only describes it.
- This repo does **not** replace `HHPE-HRG/xlotyl` or `HHPE-HRG/stoneforge` for code review, CI, or shipping.
- Edits to chunks or `generated/` files are wiped on next refresh. To change what gets mirrored, edit `context-config.json` and `refresh.sh`, not the outputs.
